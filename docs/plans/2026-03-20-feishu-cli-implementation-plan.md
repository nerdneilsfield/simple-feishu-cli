# Feishu CLI Implementation Plan

日期：2026-03-20

## 1. 目标

按已批准设计实现一个最小可用的 Feishu CLI，具备以下能力：

- 使用企业自建应用凭证认证
- 发送文本消息
- 上传本地文件并发送文件消息
- 支持命令行参数、环境变量、配置文件三种凭证来源
- 支持 GitHub Actions CI 与 Release
- 支持 `goreleaser` 多平台发布

## 2. 依赖

开始实现前需要满足以下前置条件：

- Go 版本已确定并写入 `go.mod`
- 选定官方 SDK `larksuite/oapi-sdk-go`
- 确定 CLI 框架为 `cobra`
- GitHub 仓库可用 GitHub Actions
- 发布流程以 Git tag 驱动

## 3. 步骤

### 1. 初始化项目骨架

具体行动：

- 初始化 `go.mod`
- 建立基础目录结构
- 添加 CLI 入口和根命令

验证标准：

- `go test ./...` 可运行
- 执行 `feishu --help` 能看到根命令帮助

估计工作量：

- 小

### 2. 实现配置加载模块

具体行动：

- 定义运行时配置结构
- 实现命令行参数、环境变量、配置文件加载
- 实现优先级合并逻辑

验证标准：

- 单元测试覆盖三种来源及优先级
- 缺少凭证时返回明确错误

估计工作量：

- 小

### 3. 实现飞书客户端封装

具体行动：

- 初始化官方 SDK 客户端
- 封装统一错误转换
- 定义内部发送接口

验证标准：

- 可通过单元测试验证输入构造和错误包装
- 内部接口不泄露 CLI 层细节

估计工作量：

- 中

### 4. 实现 `send text`

具体行动：

- 添加 `send` 与 `send text` 子命令
- 校验 `--to-type`、`--to`、`--text`
- 调用飞书消息发送接口

验证标准：

- 单元测试覆盖参数错误
- Mock 或接口抽象可验证正确调用发送逻辑
- 成功输出稳定字段

估计工作量：

- 中

### 5. 实现 `send file`

具体行动：

- 添加 `send file` 子命令
- 校验 `--to-type`、`--to`、`--path`
- 校验本地文件存在且可读
- 实现上传后发送的完整流程

验证标准：

- 单元测试覆盖文件不存在、参数缺失、上传失败、发送失败
- 成功路径返回 `message_id`

估计工作量：

- 中

### 6. 统一输出与退出码

具体行动：

- 定义成功输出格式
- 定义统一错误输出格式
- 为参数错误、配置错误、API 错误设定一致退出行为

验证标准：

- 命令行为在成功和失败时输出稳定
- 可被 shell/CI 稳定消费

估计工作量：

- 小

### 7. 编写 README 和示例

具体行动：

- 说明安装方式
- 说明三种凭证配置方式
- 给出文本发送、文件发送、CI/CD 示例
- 列出必要飞书权限和常见错误

验证标准：

- 新用户仅根据 README 即可完成一次发送

估计工作量：

- 小

### 8. 加入测试与基础质量门禁

具体行动：

- 为配置模块和 CLI 参数校验补单元测试
- 加入 `go test ./...`
- 视复杂度加入 `go vet ./...`

验证标准：

- 本地和 CI 中均可通过

估计工作量：

- 小

### 9. 配置 GitHub Actions

具体行动：

- 新增 `ci.yaml`
- 在 `push` / `pull_request` 上执行测试与基础校验
- 新增 `release.yaml`
- 在 tag 发布时执行 `goreleaser`

验证标准：

- PR 能自动跑 CI
- 推送 `v*` tag 能触发发布工作流

估计工作量：

- 小

### 10. 配置 `goreleaser`

具体行动：

- 添加 `.goreleaser.yml`
- 配置目标平台矩阵
- 生成 archive 和 checksum
- 发布 GitHub Release

验证标准：

- `goreleaser release --snapshot --clean` 本地可跑通
- 产物包含约定平台：
  - `linux/amd64`
  - `linux/arm64`
  - `darwin/amd64`
  - `darwin/arm64`
  - `windows/amd64`
  - `windows/arm64`

估计工作量：

- 小

## 4. 检查点

以下阶段适合做用户审查：

1. 项目骨架与 CLI 命令面完成后
2. `send text` 与 `send file` 功能完成后
3. README、CI、Release 配置完成后

## 5. 风险

### 5.1 官方 SDK 接口细节与预期不完全一致

缓解措施：

- 在实现前先确认上传文件和发送消息的具体 SDK 调用方式
- 封装层保持薄，避免过度抽象

### 5.2 文件上传接口对文件类型或权限有限制

缓解措施：

- 保留 API 原始错误信息
- 在 README 中补充约束说明

### 5.3 CI 难以做真实飞书集成测试

缓解措施：

- 第一版以单元测试和参数校验测试为主
- 将真实飞书联调作为手动验证步骤

### 5.4 Windows ARM64 发布难以做运行验证

缓解措施：

- 第一版接受构建级验证
- 后续再补充目标环境验证
